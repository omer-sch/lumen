The five views
Three workhorses, one dimension, one static reference. All live in yellowhead-visionbi-rivery.yh_app_layer.
Fact view · workhorseDaily campaign-level
fact_daily_campaign
Grain: (date, master_account_id, campaign_id, network)
Unified daily fact view across the 4 healthy platforms. Filters NULL master_account at source. Adds client_key, client_slug, normalized network labels. 90% of consumer queries hit this.
Dimension viewOne row per client
dim_client
Grain: client_key
Active UA roster joined with spend recency. Carries team, account_manager, monthly_budget, dashboard_link from the sales table. Includes computed status: active, paused, or stale.
Helper view · honestyDaily per-network
v_unattributed_spend
Grain: (date, network)
Daily spend where master_account is NULL. Surfaces the Apple unattributed problem so consumers can render an honest "X% unattributed" footnote without doing their own filter math.
Helper view · trust4 rows total
v_data_freshness
Grain: network
Per-platform latest data date. One query, four rows. Lets any consumer build a freshness banner without round-tripping through Rivery telemetry.
Reference table · static4 rows, hand-maintained
dim_network
Grain: network
Static reference for display labels, parent family grouping (Meta + Meta iOS14 → Meta family), brand color, sort order. Hand-edited by us when networks change.
What consumers can rely on
The promise to anyone reading this layer. Versioned. Stable. App-agnostic.
Schema stability
Column names and types do not change without a deprecation period. New columns appended; old ones never silently removed.
Identifier consistency
client_key is always LOWER(TRIM(master_account)). Never master_account_id.
Network normalization
Always one of: Meta, Meta iOS14, Apple Search Ads, Google Ads. No raw upstream labels leak through.
No NULL identifiers
Rows where master_account is NULL are filtered at the view. Unattributed total exposed separately in v_unattributed_spend.
Always live
Views run against source on every query. Freshness lags only the upstream management_dashboard_* refresh (~daily 09:00 UTC).
Cost baseline
Full fact_daily_campaign scan ≈ 110 MB. Date-filtered queries scale linearly. Always include a date filter.

V1

Failure modes
What can go wrong, in decreasing likelihood. All three have one-file recoveries.
Upstream column rename
A column in management_dashboard_fb2 or similar gets renamed. Views fail loudly with a clear BQ error. Recovery: edit the view DDL, redeploy. One-file change.
Upstream table dropped
Same loud failure. If replaced by dwh_management_dashboard_new, rewrite is one UNION branch per platform. One-day change at most.
Roster Team values drift
New variants of "UA" appear in the roster and dim_client silently drops them. Detection: V2 returns a number much lower than expected. Daily monitor catches it within 24h.
Not in this version of the layer
Future additions. None block Phase 1.

fact_daily_campaign_singular for Superbloom Games clients (Venue, aTable, Highrise, Obsidian Knight, Kingdom Maker, Mundo Slots) via yh_singular + pw_yh_cohort_*.
v2

v2
fact_daily_creative for adset / ad / creative grain. Requires sourcing from per-platform dwh_* and uni_* tables.
v3
fact_daily_geo, fact_daily_placement for country and placement breakdowns. Available in uni_*.
v3
fact_anomaly exposing ml_superbloom_* and metalstorm_* outputs once we understand the schema.
v4
Organic / Creative / CSM data. Not in the warehouse yet, so cannot be in the app layer.
Done means
The dataset yh_app_layer exists, contains the 5 views, all 5 validation queries pass, the lumen-app service account has read access, and the contract is documented.
After that, every consumer reads the app layer and never touches yellowhead_prod directly. The Lumen TypeScript work that follows is trivial: query the views, render the result. No union logic, no roster join, no NULL filtering, no normalization. All of that lives in BigQuery, once, where any tool can use it.
