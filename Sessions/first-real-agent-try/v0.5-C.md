**Server library**:
- `src/lib/gmail/encryption.ts` · AES-256-GCM round-trip for tokens, key from `GMAIL_TOKEN_ENCRYPTION_KEY` (32 bytes hex). Handles both base64 (Supabase wire) and `\x` hex (psql) bytea formats.
- `src/lib/gmail/oauth.ts` · hand-rolled Google OAuth2 (no SDK). Auth-URL builder, code-for-token exchange, refresh, id_token email decode.
- `src/lib/gmail/tokens.ts` · encrypted save / load, transparent refresh on read when within 60s of expiry.
- `src/lib/gmail/api.ts` · thin Gmail client. `startWatch`, `stopWatch`, `listHistory`, `getMessage`, plus pure helpers `extractMessageBody` (text/plain preferred, html stripped) and `parseFromAddress`.
- `src/lib/gmail/watch.ts` · registerWatch, loadWatch, setWatchHistoryId, markWatchFailed, stopAndDeleteWatch, listExpiringWatches.
- `src/lib/email-filters.ts` · CRUD + `emailMatchesFilters` pure helper + seedDefaultFiltersForUser (pulls contact domains).
- `src/lib/notifications/server.ts` · pushNotification, listNotifications, countUnread, markAllRead. (Co-exists with the legacy `src/lib/notifications/store.ts` client-side mock that the TopBar currently reads from; rewiring TopBar is queued.)

**Routes**:
- `GET /api/auth/gmail/start` · Clerk-authed; redirects to Google consent.
- `GET /api/auth/gmail/callback` · exchanges the code, saves tokens, seeds default filters from GlobalComix contact domains, registers the watch, pushes a `gmail_connected` notification, redirects to `/settings/integrations?ok=connected`.
- `POST /api/auth/gmail/disconnect` · stops the Gmail watch on Google's side, drops the tokens + watch row.
- `POST /api/webhooks/gmail` · public; Pub/Sub push handler. Constant-time verifies `GOOGLE_PUBSUB_VERIFICATION_TOKEN`, resolves email->user, pulls Gmail history since `gmail_watches.history_id`, filters by `user_email_filters`, fires the Hermes graph for each match, pushes a `hermes_draft_ready` notification per draft, advances the cursor. Rate-limited at 20 Hermes runs / hour / user.
- `GET /api/cron/gmail-watch-refresh` · cron-secret authed; re-registers watches expiring within 36h, pushes a `gmail_watch_failed` notification on permanent failures.
- `GET/POST /api/email-filters` · list + add filters.
- `PATCH/DELETE /api/email-filters/[id]` · toggle + delete.
- `GET /api/notifications` · list + unread count.
- `POST /api/notifications/read-all` · mark-all-read.

**UI**:

## What you (Omer) need to do before the demo path runs end-to-end

Hermes-via-Gmail needs five things outside the code: a GCP project, a Pub/Sub topic, an OAuth client, an encryption key, and a public webhook URL.

### 1. GCP project (one-time)

In the existing yellowHEAD GCP project (or a new one):
- Enable **Gmail API** and **Cloud Pub/Sub API** (`console.cloud.google.com/apis/library`).
- Create a Pub/Sub topic named `lumen-gmail-notifications`.
- Create a push subscription on that topic with the push endpoint set to your Lumen URL: `https://<your-lumen-host>/api/webhooks/gmail` (locally for testing, use ngrok or `vercel dev --listen` with a tunnel).
- Grant the system service account `gmail-api-push@system.gserviceaccount.com` the `roles/pubsub.publisher` role on the topic so Google can publish.
- On the push subscription, configure an "authentication token" (verification token); copy the value, you will paste it as `GOOGLE_PUBSUB_VERIFICATION_TOKEN`.

### 2. OAuth client (one-time)

In the same GCP project:
- Go to `APIs & Services > Credentials > Create OAuth client ID > Web application`.
- Authorized redirect URIs:
  - `http://localhost:3000/api/auth/gmail/callback` (local)
  - `https://<your-lumen-host>/api/auth/gmail/callback` (prod / preview)
- On the OAuth consent screen, request scopes `gmail.readonly` and `gmail.metadata` and add the test users who'll connect (Lior, you).
- Copy the client_id and client_secret.

### 3. Encryption key (one-time)

```
node -e "console.log(require('crypto').randomBytes(32).toString('hex'))"
```

That hex string is `GMAIL_TOKEN_ENCRYPTION_KEY`. If this key is ever lost, every connected user must reconnect (their stored tokens become unrecoverable, by design).

### 4. Env vars

Add to `.env.local` and to Vercel (preview + prod):

```
GOOGLE_OAUTH_CLIENT_ID=
GOOGLE_OAUTH_CLIENT_SECRET=
GOOGLE_PUBSUB_TOPIC=projects/<your-project-id>/topics/lumen-gmail-notifications
GOOGLE_PUBSUB_VERIFICATION_TOKEN=
GMAIL_TOKEN_ENCRYPTION_KEY=
LUMEN_APP_URL=
```

(Local dev: `LUMEN_APP_URL=http://localhost:3000`. Prod: your deployed host.)

### 5. Public webhook URL (only for local testing)

Google Pub/Sub push will not reach `http://localhost:3000`. Two options:
- Deploy to a Vercel preview and set `LUMEN_APP_URL` to the preview URL.
- Use a tunnel: `ngrok http 3000` and put the ngrok https URL into both `LUMEN_APP_URL` and the Pub/Sub push subscription endpoint. Update both as the tunnel URL changes.

## End-to-end demo path (after the prereqs)

1. Visit `/settings/integrations` -> "Connect Gmail" -> consent.
2. You land back on `/settings/integrations?ok=connected` with your address in the status block.
3. The default filters seeded from GlobalComix's contact domains (`globalcomix.com`) appear under "Watch these senders." Edit if needed.
4. From a separate inbox, send an email to your connected address from one of those domains (e.g. `emily@globalcomix.com`) with a body that asks for a weekly review.
5. Within ~5-30 seconds (Pub/Sub push latency + Hermes pipeline ~30s), a notification row exists for you: "Hermes drafted a report for GlobalComix."
6. `/agents/hermes` Recent Runs table picks up the new run; clicking "Open" lands on `/reports/<id>?source=hermes`.
7. Review + edit + regenerate + export, same as the workstream-A flow.

## Deviations from spec

- **No googleapis SDK.** Hand-rolled OAuth + Gmail fetch calls. Fewer deps; identical behaviour for the four endpoints we touch. Adding the SDK is a trivial refactor if a future feature needs `users.threads`, `users.drafts`, etc.
- **Token encryption via Node crypto, not Supabase pgcrypto.** Spec mentioned pgp_sym_encrypt; AES-256-GCM in Node keeps the key out of SQL strings and `pg_stat_statements`. Same threat model coverage with smaller blast radius on a key leak.
- **TopBar bell still reads from the v0 localStorage mock store** (`src/lib/notifications/store.ts`). The new `src/lib/notifications/server.ts` + `/api/notifications` endpoint are wired, but rewiring `TopBar.tsx` + `NotificationPanel.tsx` to the server store is queued as a UI swap. The Hermes profile's Recent Runs already surfaces new drafts; the bell is a polish item.
- **Webhook runs Hermes inline** (synchronous inside the request) rather than queuing. Pub/Sub push has a 600s ack deadline; Hermes runs in ~30s; we have headroom. A queue-and-ack split is the obvious upgrade if traffic grows.
- **PREVIEW namespacing for the user id** still uses a constant `"preview-user"` (security carry-over from workstream B). The webhook + cron paths don't run under PREVIEW so the collision risk is local-only.

## Open / queued for follow-up

- **TopBar bell rewiring** to /api/notifications + /api/notifications/read-all (currently the panel is fed by a localStorage mock).
- **OAuth state HMAC.** State is the bare userId today; the callback re-derives the user from the Clerk session and compares against state. A signed HMAC would prevent state forgery if Clerk session validation were ever bypassed.
